# 課程五：第 10 週 - 提示工程基礎：溝通的藝術 (Prompt Engineering)

給予正確的指令，AI 就是天才；指令錯誤，AI 就是庸才。本週我們將學習提示工程 (Prompt Engineering) 的核心要素，掌握如何透過角色設定、上下文填充與格式控制，精準引導大語言模型 (LLM) 產出高品質結果。

---

## 一、 單元講解 (Lecture) - 100 分鐘

### 1. 提示工程 (Prompt Engineering) 的定義
- **本質**：將人類意圖轉化為模型最易理解的文字指令。
- **目標**：提高輸出準確度、穩定格式、減少幻覺。

### 2. Prompt 的六大核心要素 (CO-STAR 框架)
- **C (Context)**：背景資訊。提供任務發生的環境。
- **O (Objective)**：目標。明確要求模型做什麼。
- **S (Style)**：風格。指定寫作風格（如：幽默、專業）。
- **T (Tone)**：語氣。設定情感基調（如：親切、嚴肅）。
- **A (Audience)**：讀者。誰會看到這份結果？
- **R (Response)**：格式。指定輸出 JSON, Table 或 Markdown。

### 3. 提示技巧：Zero-shot 與 Few-shot
- **零樣本 (Zero-shot)**：直接下指令（適合簡單、常識性任務）。
- **少樣本 (Few-shot)**：給予 1-3 個範例（適合複雜邏輯或特定格式要求）。

### 4. 角色設定 (Persona) 的威力
- **原理**：透過設定「你是資深架構師」或「你是五歲小孩」，模型會自動調用關聯的語義空間，調整專業深度。

### 5. 指令分隔符與負面約束
- **分隔符**：使用 `###` 或 `---` 區分指令與原始資料，防止模型混淆。
- **約束**：明確告訴模型「不要做什麼」（如：不要使用專業術語）。

---

## 二、 動手實作 (Lab) - 50 分鐘

### 任務：Prompt 優化大挑戰
1.  **從模糊到精準**：
    - **原始指令**：寫一篇關於 AI 的文章。
    - **任務**：將其修改為包含角色、背景、目標讀者與字數限制的高品質 Prompt。
2.  **格式化控制賽**：
    - 撰寫一個 Prompt，輸入 5 位員工的打卡時間，要求模型以 JSON 格式輸出其工時統計。
    - 驗證模型輸出的 JSON 格式是否正確。
3.  **Few-shot 實踐**：
    - 任務：將白話文翻譯成特定的「江湖語氣」。
    - 步驟：提供 2 個江湖範例，然後輸入新句子觀察生成效果。
4.  **反向 Prompt 練習**：
    - 展示一段高品質的 AI 生成內容，讓學生嘗試推導出「背後的原始 Prompt 是什麼」。

## 三 island 知識點回顧 (KP)

- **[KP 5.10.1] Prompt 結構化思維**：掌握背景、任務、輸出格式三位一體的撰寫習慣。
- **[KP 5.10.2] 角色設定的心理學效應**：理解 Persona 如何限縮模型的生成範圍。
- **[KP 5.10.3] 少樣本學習的應用時機**：學會透過範例解決模型無法理解的抽象格式要求。
- **[KP 5.10.4] 輸出格式鎖定技術**：明白如何強制模型產出機器可讀（JSON/CSV）的格式。
- **[KP 5.10.5] 提示詞迭代策略**：掌握「觀察 -> 修改 -> 測試」的螺旋式優化流程。

## 四、 課後測驗題庫 (Quiz) - 30 分鐘

### 1. 單選題 (Single Choice) - 10 題

1. **Q1**: 提示工程 (Prompt Engineering) 最核心的目的在於？
   - (A) 增加模型參數 (B) 提高與 AI 溝通的效率與產出品質 (C) 加快硬體運算 (D) 刪除多餘數據
2. **Q2**: 「你是具備 20 年經驗的律師」，這在提示語中屬於哪一部分？
   - (A) 目標 (Objective) (B) 角色設定 (Persona) (C) 負面提示 (Negative) (D) 分隔符
3. **Q3**: 若希望 AI 輸出可以直接被 Excel 讀取，最建議要求的格式是？
   - (A) JSON (B) CSV 或 Table (C) Markdown (D) 圖檔
4. **Q4**: 「不給予任何範例，直接讓模型執行任務」的方式稱為？
   - (A) Zero-shot (B) Few-shot (C) Multi-shot (D) One-shot
5. **Q5**: 在 Prompt 中使用 `"""` 或 `###` 的主要作用是？
   - (A) 為了好看 (B) 區分指令與內容，防止指令注入混淆 (C) 增加 Token 數量 (D) 加密內容
6. **Q6**: 關於 Temperature (溫度) 參數，若希望 AI 輸出「嚴謹、穩定且固定」的結果，應調？
   - (A) 高 (如 1.2) (B) 低 (如 0.2) (C) 不變 (D) 關閉
7. **Q7**: 下列哪種技巧最適合引導模型處理「從未見過的新規則」？
   - (A) 增加感嘆號 (B) 提供 Few-shot 範例 (C) 使用大寫字母 (D) 換一個模型
8. **Q8**: 「不要使用任何技術術語，讓五歲小孩也能聽懂」，這屬於？
   - (A) 背景 (B) 約束與風格 (Constraint/Style) (C) 數據來源 (D) 角色
9. **Q9**: 提示詞中，將字數限制寫為「約 300 字」比「寫一篇長文」更好的原因是？
   - (A) Token 比較少 (B) 指令更具體明確 (C) 運算比較快 (D) 節省電力
10. **Q10**: 提示工程師在優化 Prompt 時，最應該具備的循環思維是？
    - (A) 猜測 -> 放棄 (B) 測試 -> 反饋 -> 修正 (C) 複製 -> 貼上 (D) 讀取 -> 儲存

### 2. 多選題 (Multiple Choice) - 10 題

11. **Q11**: 高品質的 Prompt 通常應包含哪些部分？
    - (A) 明確的指令 (B) 充足的背景資料 (C) 預期的輸出格式 (D) 限制條件
12. **Q12**: 角色設定 (Persona) 的好處有？
    - (A) 鎖定特定領域的專業詞彙。 (B) 調整回覆的語氣。 (C) 降低幻覺機率 (某種程度)。 (D) 讓 AI 變得有靈魂。
13. **Q13**: 關於 Few-shot 提示，正確的有？
    - (A) 範例的質量比數量更重要。 (B) 範例應包含輸入與期望的輸出。 (C) 範例越長越好。 (D) 適合用來教導模型特定的排版格式。
14. **Q14**: 指令注入 (Prompt Injection) 的風險包括？
    - (A) AI 洩露不該說的敏感資訊。 (B) 用戶繞過安全過濾器。 (C) 模型被引導執行惡意代碼。 (D) 電腦中毒。
15. **Q15**: 在設計業務流程 AI 時，為什麼格式控制 (JSON/YAML) 很重要？
    - (A) 方便後續程式自動化處理。 (B) 確保數據結構穩定。 (C) 增加美觀。 (D) 減少人工解析成本。
16. **Q16**: 哪些符號常用作 Prompt 中的分隔符？
    - (A) `###` (B) `---` (C) `"""` (D) `[[ ]]`
17. **Q17**: 提示工程對企業的價值在於？
    - (A) 提升 AI 助理的可用性。 (B) 降低對後端代碼重寫的依賴。 (C) 加速產品原型開發。 (D) 取代所有程式設計師。
18. **Q18**: 迭代優化 Prompt 的步驟應包含？
    - (A) 觀察模型的錯誤點。 (B) 修改不具體的指令。 (C) 測試邊界案例。 (D) 換用不同的參數設置。
19. **Q19**: 在 Prompt 中加入「這對我來說非常重要，請認真對待」等情感詞，某些研究認為？
    - (A) 完全沒用。 (B) 可能在特定大模型中提升輸出質量。 (C) 增加 Token 浪費。 (D) 會導致模型死機。
20. **Q20**: 指令設計中的常見錯誤？
    - (A) 指令太模糊。 (B) 指令之間相互衝突。 (C) 給予太多無關的背景資訊。 (D) 要求模型做它不具備的能力（如：預測未來）。
