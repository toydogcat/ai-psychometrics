# 課程五：第 14 週 - 低代碼/無代碼 AI 解決方案開發 (No-code AI)

不寫程式也能建構 AI 應用？本週我們將認識「無代碼 (No-code)」與「低代碼 (Low-code)」開發工具。你將學會如何透過拖拉拽的方式，將 AI 模型、資料庫與自動化流程串接起來，快速產出商業原型 (MVP)。

---

## 一、 單元講解 (Lecture) - 100 分鐘

### 1. 為什麼需要無代碼 AI 開發？
- **速度**：從想法到上線只需幾小時而非數週。
- **門檻**：讓業務人員 (PM/Planner) 也能直接參與技術實現。
- **維護**：視覺化介面降低了理解與交接的難度。

### 2. OpenAI GPTs 與 個人化助理
- **自定義指令 (System Prompt)**：賦予助理特定行為。
- **知識庫掛載**：讓助理只回答你上傳的 PDF 內容（簡易 RAG）。
- **能力組合**：聯網、繪圖、代碼執行器的開關。

### 3. 自動化工作流串接：Make.com 與 Zapier
- **觸發器 (Trigger)**：事件的起點（如：收到一封 Gmail）。
- **行動 (Action)**：AI 的介入（如：讓 GPT 總結郵件並存入 Google Sheet）。
- **邏輯分叉**：如果情感是負面的，則通知主管；如果是正面的，則自動回信。

### 4. 低代碼平台概覽
- **Microsoft Copilot Studio**：企業級對話機器人開發。
- **Coze (扣子)**：強大的外掛 (Plugins) 生態與多代理 (Multi-agent) 協作。

### 5. MVP (最小可行性產品) 思維
- **規劃**：不要一開始就想做完美的 App，先解決一個核心痛點。
- **成本**：學會估算第三方工具的月租與 API 消耗。

## 二、 動手實作 (Lab) - 50 分鐘

### 任務：打造你的自動化 AI 助理
1.  **GPTs 實戰**：
    - 在 OpenAI 平台上建立一個「ipas 認證助手」。
    - 上傳本課程的大綱文件作為知識庫。
    - 測試模型是否能根據文件內容為你出一道考題。
2.  **自動化流程設計 (以 Make 或 Zapier 模擬)**：
    - 繪製一個流程圖：當客戶在 Google Form 提交問題 -> AI 自動分類 -> 存入 Notion 資料庫。
3.  **多代理協作體驗 (Coze)**：
    - 在 Coze 平台建立一個機器人。
    - 學習如何添加「Google 搜尋」與「DALL-E 繪圖」插件。
    - 測試機器人是否能幫你「查新聞並配圖」。
4.  **成本試算表**：
    - 假設每月有 1000 封郵件需要 AI 處理。
    - 計算選用 OpenAI API vs. 買專用軟體的成本差異。

## 三 island 知識點回顧 (KP)

- **[KP 5.14.1] 無代碼開發的價值鏈**：理解其在快速驗證商業假說中的關鍵角色。
- **[KP 5.14.2] 指令、知識與能力的配置**：掌握構建自定義助理的三大支柱。
- **[KP 5.14.3] API 驅動的自動化邏輯**：明白 Webhook、觸發器與行動的串接原理。
- **[KP 5.14.4] 插件 (Plugins) 的延伸功能**：掌握如何透過外部工具補足 LLM 的能力。
- **[KP 5.14.5] 解決方案的技術債**：識別無代碼工具在擴展性與數據安全性上的潛在局限。

## 四、 課後測驗題庫 (Quiz) - 30 分鐘

### 1. 單選題 (Single Choice) - 10 題

1. **Q1**: 下列哪一個術語最適合描述「不需要撰寫複雜程式碼，透過視覺化介面開發應用」？
   - (A) Full-stack (B) No-code (C) Hard-coding (D) Assembly
2. **Q2**: OpenAI 提供的「GPTs」功能，其核心價值在於？
   - (A) 讓用戶自定義特定用途的 AI 助手 (B) 提高網速 (C) 增加 CPU 核心數 (D) 永久免費
3. **Q3**: 在自動化工具 Make.com 或 Zapier 中，流程的起點通常被稱為？
   - (A) Action (行動) (B) Trigger (觸發器) (C) Result (結果) (D) Loop (迴圈)
4. **Q4**: 想要為你的自定義 AI 助手加入「公司最新的產品手冊」，應使用哪種功能？
   - (A) 更改背景顏色 (B) 掛載知識庫文件 (C) 增加字體大小 (D) 修改 API Key
5. **Q5**: 相比於從頭編寫代碼，低代碼工具最大的缺點通常是？
   - (A) 學習曲線太高 (B) 開發速度太慢 (C) 靈活性與自定義深度受到平台限制 (D) 畫面太醜
6. **Q6**: 「當我收到一封帶有附件的郵件，自動將附件轉傳給 AI 進行分析」，這是一個典型的？
   - (A) 數據庫查詢 (B) 自動化工作流 (Workflow) (C) 硬體中斷 (D) 排序演算法
7. **Q7**: 哪個平台以提供豐富的 AI 「外掛 (Plugins)」與 多代理協作著稱？
   - (A) Excel (B) Coze (C) Notepad (D) WinRAR
8. **Q8**: 在規劃 AI 應用時，所謂的「MVP」是指？
   - (A) 最有價值的球員 (B) 最小可行性產品 (Minimum Viable Product) (C) 最大運算效能 (D) 多版本控制
9. **Q9**: 使用無代碼工具串接不同服務（如 LINE 和 ChatGPT）時，最常用的技術介面是？
   - (A) 藍牙 (B) API (C) 撥接 (D) 傳真
10. **Q10**: 誰最適合擔任「無代碼 AI 解決方案」的開發者？
    - (A) 只有資深工程師 (B) 具備業務理解能力的產品經理或規劃師 (C) 只有硬體維修員 (D) AI 模型本身

### 2. 多選題 (Multiple Choice) - 10 題

11. **Q11**: 無代碼/低代碼工具對企業的優點？
    - (A) 降低 IT 部門的壓力。 (B) 加速數位轉型。 (C) 方便非技術人員進行創新實驗。 (D) 完全不需要支付任何軟體費用。
12. **Q12**: 在建構自定義 GPTs 時，你可以配置哪些選項？
    - (A) 系統指令 (Instructions)。 (B) 檔案上傳 (Knowledge)。 (C) 操作功能 (Actions/API)。 (D) 模型的原始神經元權重。
13. **Q13**: 哪些工具屬於常見的「自動化工作流」平台？
    - (A) Zapier (B) Make.com (C) Microsoft Power Automate (D) Spotify
14. **Q14**: 為什麼在規劃 AI 應用時，先做原型 (Prototype) 很重要？
    - (A) 驗證用戶需求是否真實。 (B) 測試 AI 處理該問題的準確度。 (C) 節省後續大規模開發的風險。 (D) 為了向投資人展示實力。
15. **Q15**: 關於 API 的 Request 與 Response，正確的有？
    - (A) Request 是發送給 AI 的請求。 (B) Response 是 AI 回傳的結果。 (C) 兩者之間不需要網路。 (D) 格式通常為 JSON。
16. **Q16**: 無代碼工具在安全性上的考量？
    - (A) 敏感數據是否會被第三方工具截留。 (B) API Key 是否妥善保存。 (C) 模型是否會產生過度隨機。 (D) 電腦是否會爆炸。
17. **Q17**: 低代碼 AI 平台的發展趨勢？
    - (A) 整合越來越多的 AI 插件。 (B) 操作越來越簡單（自然語言對話式開發）。 (C) 支援多代理 (Multi-agent) 協作。 (D) 最終會消失被手寫代碼取代。
18. **Q18**: 構建一個「AI 閱讀助手」需要哪些步驟？
    - (A) 設定角色為「閱讀專家」。 (B) 上傳目標文件。 (C) 設定輸出的格式要求。 (D) 自行訓練一個新的 LLM。
19. **Q19**: 哪些是適合無代碼 AI 導入的場景？
    - (A) 自動過濾與回覆簡單的客戶諮詢。 (B) 定期將會議記錄摘要並發送通知。 (C) 處理具備機密軍事數據的運算。 (D) 快速產出社群媒體文案。
20. **Q20**: 面對眾多的無代碼工具，規劃師應如何選型？
    - (A) 考慮成本 (Pricing)。 (B) 考慮與現有工具（如 Google/Slack）的相容性。 (C) 考慮數據處理的額度限制。 (D) 隨機選一個。
