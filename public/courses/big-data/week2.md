# 課程六：第 2 週 - 數據採集、ETL 與 流水線自動化 (ETL Pipelines)

數據採集與轉換是 AI 專案中耗時最長卻最重要的基礎。本週我們將詳解 ETL (Extract, Transform, Load) 流程，學習如何使用 Apache Airflow 進行任務排程，並探討實時數據流與批次處理的架構權衡。

---

## 一、 單元講解 (Lecture) - 總計 100 分鐘

### 1. ETL 流程深度拆解 (20 min)
*   **Extract (提取) (KP2.1)**：從多樣化數據源 (API, Logs, SQL, NoSQL) 獲取原始數據。需要考慮增量提取 (Incremental) 與全量提取。
*   **Transform (轉換) (KP2.1)**：數據處理的核心。包含數據清洗 (去重、修正異常)、格式轉換、彙整計算與特徵初步提取。
*   **Load (載入) (KP2.1)**：將處理後的「乾淨」數據寫入目標系統（如 Data Warehouse 或 Feature Store），供後續模型訓練或推理使用。

### 2. 數據規範化與任務相依性 (20 min)
*   **數據規範化 (Normalization) (KP2.2)**：減少冗餘，確保數據一致性。反規範化 (Denormalization) 則常用於提升大數據查詢效率。
*   **DAG (有向無環圖) (KP2.3)**：定義任務之間的先後順序與相依關係。例如：必須先執行「數據採集」才能執行「數據清洗」。
*   **任務相依性管理 (KP2.3)**：如何處理任務失敗、重試機制 (Retry) 以及任務間的參數傳遞 (XComs)。

### 3. 流水線自動化：Apache Airflow (20 min)
*   **架構組件 (KP2.5)**：Scheduler (調度器)、Web Server (監控介面) 與 Worker (執行單元)。
*   **代碼即工作流 (Workflow as Code)**：使用 Python 定義 DAG，便於版本控制 (Git) 與團隊協作。
*   **監控與告警 (KP2.5)**：即時掌握流水線狀態，當 ETL 失敗時第一時間通知工程師。

### 4. 數據架構選型：Lambda vs. Kappa (20 min)
*   **Lambda 架構 (KP2.4)**：包含批次層 (處理歷史海量數據) 與 實時層 (處理最新數據)。優點是準確性高，缺點是維護兩套代碼邏輯。
*   **Kappa 架構 (KP2.4)**：將所有數據視為流 (Stream)，簡化為單一處理層。適合對實時性要求極高且處理邏輯一致的場景。
*   **Kafka 在攝取中的角色 (KP2.4)**：作為高吞吐量的訊息緩衝層，平衡數據生產速度與處理速度的差異。

### 5. 資料血緣 (Data Lineage) 與治理 (20 min)
*   **資料血緣 (KP2.5)**：追蹤數據從源頭到終點的所有變更路徑。對於故障診斷與法規稽核至關重要。
*   **元數據管理 (Metadata) (KP2.5)**：維護「關於數據的數據」，如欄位定義、數據擁有者、敏感度等級。
*   **數據質量監控**：在流水線中加入檢查點 (Checkpoints)，防止垃圾數據污染特徵庫。

## 二、 動手實作 (Lab) - 總計 50 分鐘

### 實作一：Python 自動化數據清理腳本 (20 min)
**任務目標**：實作 ETL 中的 Transform 邏輯。
1.  讀取包含噪聲的原始數據。
2.  撰寫邏輯：刪除重複列、將日期格式統一、填補缺失的數值欄位（使用中位數）。
3.  輸出清理後的 CSV 檔並驗證數據分布。

### 實作二：使用 Airflow 定義簡單任務流 (30 min)
**任務目標**：理解 DAG 的基本定義方式。
1.  建立一個 Python 檔案定義 DAG。
2.  設置三個 Task：`extract_task`, `transform_task`, `load_task`。
3.  使用 `>>` 符號定義依賴關係：`extract_task >> transform_task >> load_task`。
4.  模擬其中一個 Task 失敗，觀察 Airflow 的重試行為。
